    LLVM Execution and Interpretation : LLVM Execution Engine, Just-In-Time Compiler and Interpreter

    Skip to end of metadata
    • Created by Trang Tran Thi Thuy, last modified on Feb 27, 2015
    Go to start of metadata

    LLVM Architecture Quick Reminder

    LLVM is designed with flexibility in mind. It splits the compiling process into three parts: Front end, Optimizer and Back-end supporting both multiple source languages and target architectures. The general ideas is to transform multiple different source languages into one common code representation, aka LLVM IR. After the LLVM-IR form of the program is optimized in the optimizer, it is, later, compiled into any target language in a back end, as shown in Figure 1. With this three-phase design, adding a new target in LLVM requires only writing the back-end while the front-end and optimizer are reusable. 

    LLVM Execution Engine

    LLVM introduce Execution Engine to perform the execution of LLVM modules. It is designed to support both interpretation and Just-In-Time complier implementation. The ExecutionEngine class which is defined in <llvm_source>/include/llvm/ExecutionEngine/ExecutionEngine.h is responsible for managing the execution of an entire guest program, analyzing the next program fragment that need to run and performing necessary actions to execute it. In general, JIT engine works by writing binary blobs to the memory which are accomplished by ExecutionEngine class. Afterwards, these instructions can be executed by jumping to the allocated memory area with the function pointer that ExecutionEngine provides. LLVM provides the following execution scenarios:

    • Lazy compilation: Initially, the function is compiled whenever a pointer to its instruction is requested. With lazy compilation, the function's compilation is postponed until it is called. 
    • Compilation of external global variables: This allows symbol resolution and memory allocation of symbol outside the current LLVM module.
    • Look up and symbol resolution for external symbols via dlsym: similar to loading dynamic shared objects at runtime.

    LLVM supports three execution engine implementation: llvm:MCJIT, llvm:orc:OrcMCJITReplacement and llvm:interpreter. However, there was another implementation of ExecutionEngine, namely llvm:JIT which is removed after LLVM 3.5. 

    LLVM Just-In-Time Execution Engine

    LLVM JIT implementation of ExecutionEngine is the oldest engine and is implemented by using different parts of LLVM code generator. JIT ExecutionEngine is mostly target-independent, each target must implement the binary instruction emission for its specification. It works by first emitting the binary blobs into memory using the JITCodeEmitter class, a MachineCodeEmitter subclass. The JIT ExecutionEngine also ultilizes  memory manager, JITMemoryManager, and  a resolver, JITResolver. While the former is responsible for managing memory usage such as assigning memory for global variables, or allocating memory marked as read/write executable to emit instructions to; the latter is responsible for keep a track and resolve call sites to functions that are not yet compiled.

    To implement JIT compiler for a specific target the following class needs to be implemented:  XXXCodeEmitter and XXXJITInfo. The former contains a machine function pass that transform target-machine instruction into relocatable machine code. The latter implements a JIT interfaces for target-specific code-generation activities such as code emission and stubs.

    LLVM Machine Code Just-In-Time Execution Engine

    This ExecutionEngine ultilizes the MC (Machine Code) framework which tries to integrate assembler, disassembler and object linker into LLVM. Unlike JIT Execution Engine, which operates on function level, MCJIT ExecutionEngine compiles the whole module, meaning the whole module is JIT-compiled prior to any function execution. Since the engine uses the MC framework which shares informations about the target across the MC framework. 

    The code generation takes place when loading a Module object. The process follows the following steps. First, it creates an ObjectBuffer instance to hold a Module object. If the Module is already compiled, denoted by a state loaded, the ObjectCache interface is used to retrieve to avoid re-compilation. In case the object is not compiled, LLVM emits MC code resulting in an ObjectBufferStream. The RuntimeDyld dynamic linker loads the resulting ObjectBuffer object and builds a symbol table resulting an ObjectImage object. Finally, the Module is finalized meaning relocation are resolved, loaded modules are moved to a finalized module group and read/write memory permission are set. The Module is now ready for execution.

    Examples of LLVM Execution Engine

    LLVM provides an interpreter tool lli which is capable of interpreting LLVM bitcode and JIT compiler using both aforementioned Execution Engine.

    #include <stdio.h>
    int sum(int a, int b) {
         return a + b;
    }
     
    int main() {
         printf("sum: %d\n", sum(2, 3) + sum(3, 4));
         return 0;
    }  

    We can obtain the bitcode file by using clang

    Code Block
    ?
    clang -emit-llvm -c sum.c -o sum.bc

     

    LLVM JIT Execution Engine

    We can execute the bitcode file using the JITExecutionEngine by simply tyipng

    Code Block
    ?
    lli sum.bc
    sum: 12

     

    LLVM MCJIT Execution Engine
